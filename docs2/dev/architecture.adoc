== Key Design Patterns

=== Data Ingestion Pipeline

----
Raw Data вҶ’ Decoder вҶ’ Validation вҶ’ QC Checks вҶ’ Database вҶ’ Aggregation
----

1. *Raw data* arrives via FTP, LRGS, or manual upload
2. *Decoder* parses format-specific structure
3. *Validation* checks data types and required fields
4. *QC Checks* apply range, step, persistence tests
5. *Database* stores raw measurements
6. *Aggregation* computes hourly/daily/monthly summaries

=== Quality Control Hierarchy

----
Check Individual Limits
         вҶ“ (not found)
Check Reference Limits
         вҶ“ (not found)
Apply Global Limits
----

Station-specific limits take precedence over reference and global defaults.

=== Permission Framework

----
Request вҶ’ Middleware вҶ’ Permission Check вҶ’ View
              вҶ“
    AttachWxPermissionsMiddleware
              вҶ“
    WxGroupPageAccess model
----

Custom middleware attaches permission objects to each request based on user's groups.

=== Caching Strategy

* *Session caching* - User sessions in Memcached
* *Query caching* - Expensive queries cached
* *View caching* - Cache middleware for HTTP responses
* *Summary caching* - Pre-computed statistics

== Database Design

=== TimescaleDB Hypertables

Time-series data stored in hypertables for:

* Automatic partitioning by time
* Efficient range queries
* Data retention policies
* Compression

=== Production Considerations

* Reverse proxy with SSL termination
* External database for high availability
* Separate worker nodes for scaling
* Load balancer for multiple API instances
* Monitoring and alerting

== Security Architecture

=== Authentication Layers

1. Nginx (optional basic auth)
2. Django authentication
3. DRF permission classes
4. Custom wx permissions

=== Data Protection

* Fernet encryption for sensitive fields
* Environment-based secrets
* CORS configuration
* CSRF protection

=== Network Security

* Container network isolation
* Port exposure only via Nginx
* Internal service communication
